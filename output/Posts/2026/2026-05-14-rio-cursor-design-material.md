# 别再只画 Figma 了：我在 Cursor 看到的设计新材料

我是 Rio，现在在 Cursor 做设计负责人。过去一年，我越来越强烈地感觉到一件事：如果你真正想做软件，视觉当然重要，但它只是一层。

设计不是把像素拖到正确的位置。设计更像是在进化一组概念：它们怎么发展，每一步变成什么样子，概念之间的结构有没有更简单，每个概念有没有发挥到它最大的可能性。

有些东西堵住了，你要把它解开。有些东西本来应该是一样的，但现在不一样，你要把它 unify。有些东西本来可以更简单，你要把它简化。

这也是为什么我觉得，AI 时代设计师真正要学的，不只是怎么写 prompt，而是怎么重新理解软件的材料。

## Figma 很有用，但它不是真的

Figma 当然是好工具。Canvas 的好处也很明显：你可以很快地创建静态状态，duplicate 一个 artboard，改一点细节，看 option A、B、C、D 哪个更好。

但它的问题也在这里：**它不是真正跑起来的东西。**

它没有真实数据，没有真实逻辑，不能把一个 AI model 的 input pass 进去，不能让 agent 真的去帮你改状态，也不能让你在滚动、加载、失败、权限变化、上下文变化里看到一个界面怎么活着。

所以我说它很浅，不是说它没价值，而是说它卡在一个抽象层里。它卡在 Canvas，卡在 `.fig` 文件格式里。你画出来的那一刻，它就已经不是真的了。

我现在越来越多地直接在代码里做设计。去年刚开始，我大部分时候还是从 Framer 或 Figma 开始。现在我会直接在我们内部搭的第三版 baby Cursor 里玩。它是一个 Electron app，有 Cursor 里的 state 和 view，背后是 Cursor CLI，可以跑真的 chat。

这个东西可以很小，也可以很大。它可以非常简单，也可以非常复杂，取决于用户想要什么。我们大概花了两周做出这个 prototype。如果用传统 Figma 把所有状态都画出来，我估计要两三个月。如果再加上讨论，时间会更恐怖。

差别不只是快。更重要的是，当你 manipulate 的材料变成代码，这个东西会越来越像它本来应该有的样子。它能活起来。

## 软件不是页面，是一坨概念和它们之间的关系

我不太喜欢只从“用户想做什么”开始理解设计。那当然重要，但还有另一种方式：从软件本身看它。

软件在我眼里是一坨概念，和概念之间的关系。

最低层可能是数据结构和 infra 怎么 scale。再往上一点，是数据怎么跟界面流通。再往上一点，是界面怎么 render。最后用户看到的是 pixel。但每一层其实连着同一组概念。

你看 TikTok，它很大，但底层可以理解成 list of videos。Instagram 是 posts、stories、reels、messages。Notion 是 blocks、pages、databases。Cursor 是 agents、editors、models、tools、context。

很多 app 一开始会被自己最初设下的概念卡住。用户需求变复杂，产品也变复杂。你要么重构原有系统，要么继续往上加东西。继续加很容易，结果就是大家都想变成 super app，最后按钮越来越多，人进来反而不知道自己该干嘛。

好的设计不是“再加一个功能”。好的设计常常是看见这个功能应该插在哪里：是建一个新概念，还是改一个旧概念，还是把几个旧概念合并起来。

你不能只看一个局部 UI。你要看它和 whole system 的关系，看它和已有 pattern、已有规则、已有结构的关系。看得越清楚，真正可选的解反而越少。

## AI 让设计师绕过一些门槛，但没有绕过判断力

很多设计师现在遇到的困难，不是他们不聪明，而是软件开发前面挡着太多概念。

你打开 Cursor，看到 open project、clone repo、connect to SSH。工程师知道这是什么意思，很多设计师不知道。npm、Node、Git、branch、pull request、terminal、localhost、port，这些东西以前都是进入代码世界之前必须跨过的门槛。

理论上，agent 可以帮你绕过很多门槛。你可以先做东西，再慢慢理解这些事实。我们要做的也是把入口清理一下、翻转一下，让人先进去做一点东西，然后很快感觉到：我可以做这个，我看到的是不一样的东西。

但这不等于一个 prompt 就能完美。你需要练。你要跟不同模型玩很多东西，知道它哪里好、哪里不好。你要用 plan mode 把模糊需求展开，review 它，加更多细节。你要把 Figma link、PRD、已有 design system 放进去。你要开始给 agent 设限制，定义什么是好，定义它应该 reference 哪些东西。

设计师过去在二维空间里很容易做 variation：Command D 一个 artboard，继续改。跟 agent 玩的时候，每个 turn 都要等一会儿。如果你不理解 version control，不理解 checkpoint，不知道怎么分支，你会觉得这个反馈没有 Figma 快。

真正的变化不是“设计师都要变工程师”。变化是每个人都能从不同的层进入同一个 codebase，用同一个 agent、同一份 context 做东西。有人更在乎大方向，有人更在乎视觉细节，有人更在乎 flow，有人更懂技术。角色会继续存在，但边界会松动。

## Taste 不是装饰，它是你对系统不舒服的身体反应

很多人把 taste 理解成“这个东西漂亮不漂亮”。我觉得不够。美当然有 subjective 的部分，但它也有一些共同的东西。一个 configuration 放在一起，fit 成某种样子，你会感觉它舒服、美、简单、接近它本人的样子。

我自己看到不舒服的东西，会有很身体的反应。以前我们公司做过一个骰子，我第一反应就是去看它三个面相加对不对，因为骰子两面相加应该是 7。后来其他数据科学家也在下面留言说，对，这个加起来是对的。

这听起来像小事，但设计很多时候就是这种小事。比如中文断句、heading 的层级、badge 和按钮怎么展示、图片怎么 crop、空间之间的规则、颜色使用的规则。普通人不一定说得出来，但他们能感觉到。它像音乐，有韵律，有时候大，有时候小，但中间有规则。

AI 会让更多人做出东西，但它不会自动给你这种判断力。你还是要有眼睛，能发现细节。你也要有技能，能把这个韵律做到位。视觉层面没有问题，是最低的 bar。

再往上，有些人会更擅长概念层面的思考，有些人更擅长 flow，有些人更擅长 motion，有些人特别懂技术。一个好的团队不需要所有人都有同一种 system thinking。太多系统和系统打架，反而会僵住。

## 开始做东西，然后去生活

如果我只能给设计师一个建议，就是开始做事，用 Cursor，用代码玩一玩。

一开始会怪，会害怕，这很正常。有什么问题就问 agent。不要觉得第一次放进去它就会做出你想要的样子。它需要练，你也需要练。

但只学工具还不够。你还要去生活，去发现细节，去看，去体会。音乐、摄影、建筑、生物，都有共性。很多我们现在做的事情，过去的人其实都想过，只是材料变了。

我学 Bio 对我有帮助，因为自然系统也是一层一层的概念：底层是什么，反馈回路是什么，input 是什么，往上又怎么形成更大的结构。软件也是我们自己造出来的抽象系统。你不能卡在某一层里，你要试着理解整个东西怎么运作，怎么被架构出来，每一层之间的关系是什么，用户想要什么，公司想要什么，系统本身想要什么。

我越来越相信，设计不是一个僵硬的最终稿。它更像一个 living life form。

你不是把 Figma 画到最后一版，然后交出去。你是在让一个系统继续生长。你要看它哪里堵住了，哪里变形了，哪里可以被统一，哪里应该删掉。

你不是在搓像素。你是在进化概念。
