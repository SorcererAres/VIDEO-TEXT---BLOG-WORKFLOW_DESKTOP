# 知道什么是好的，比能写出好的更难：Agent 时代我重新理解程序员

我最近越来越强烈地感觉到一件事：程序员这个职业没有消失，但它的重心正在被整个抬高。

过去我们说一个人会写代码，通常指的是他能一行一行把程序写出来，能理解系统怎么搭，知道 frontend、backend、数据库、网络、模块、对象、过程之间如何连接。这个能力当然还重要，尤其是对少数最顶尖的人。但对大多数软件工作来说，AI coding 已经把执行层的门槛大幅降低了。

我自己是转码出身。本科读经济学，第一份工作做投行，后来因为觉得很多事情都应该自动化，才回到 Berkeley 读 Data Science。之后我去了 Meta，做过 marketplace search 和 Instagram 的 friends recommendation。真正让我开始严肃看待 AI，是后来和朋友办 AI conference。注意力一旦放过去，就会发现变化不是线性的。

从 2024 年底到现在，不过一年多一点，但体感像过了十年。我们从聊 o1、推理时计算、强化学习，一路到了 Agent、Coding Agent，再到能连续工作四五个小时的自动体。这个速度太快了。

## 程序员不再只是写代码的人

如果说过去的软件工程师是在文件里写代码，那么现在很多时候，我们是在面向 Agent 编程。

我不是说人类写的每一行都不重要了。顶尖程序员写出来的东西，仍然可能更优美、更精简、更安全。AI 生成的 code 里也会有漏洞、slop 和大量需要清理的东西。但总体上，很多 coding 工作的效率已经被推高了。哪怕对很强的程序员来说，用上 Claude Code、Codex 或类似工具，也会改变他的工作方式。

更关键的是，程序员未来不只是 line by line 去实现功能，而是要设计一套 philosophy。你要告诉系统：什么可以做，什么不能做；怎样的架构是可持续的；什么样的推荐机制是有益的，什么样的机制会长期伤害用户；什么样的数据流、安全边界、交互方式和评价指标，才符合这个产品的目标。

这就不只是写代码了。它更像一种治理。

如果一个 Agent 可以根据你的指令自己生成方法、调用工具、写文件、修 bug、跑测试，那么人类的责任就从“亲手做每一步”变成“定义好什么是对的，并且知道如何检查它”。这听上去很高级，但也更危险。因为知道什么是好的，比能写出好的更难。

## “我知道什么是好代码”不是一句轻松的话

最近有一句话很流行：我不会写代码，但我知道什么是好代码。

我同意这句话的方向，但我不觉得它轻松。因为“知道什么是好的”本身非常难。

比如推荐算法。你说最多人点赞就是好，那平台会逐渐长出一种内容生态。你说停留时间最长就是好，它又会长出另一种生态。你说公平、健康、有益、长期价值，这些词都很好听，但落到算法和指标上，每一个都会带来不同的后果。

再比如 AI hallucination。模型说错的时候，你有没有能力知道它错了？在我熟悉的领域里，我还能凭直觉判断。但如果进入医疗、生物、pharmaceutical 这类高风险领域，模型给了一个剂量、一个实验设计、一个适应人群判断，我如果没有专业经验，可能完全看不出来哪里危险。

所谓直觉，其实不是玄学。它是你一步一步做过、犯过错、修过错、被系统反复教育过之后，长出来的内部模型。问题在于，如果下一代人没有经历过一步一步搭系统的过程，他怎么长出这种直觉？

开车技能可以 degrade，因为自动驾驶越来越好。但思考技能不能 degrade。如果我们没有亲手做过，没有在具体材料里磨过，只剩下空泛的 reasoning，那很容易被模型带偏。

## 顶尖专家会更重要，但不会人人都成为专家

我倾向于认为，未来不是不需要专家，而是更需要顶尖专家。

这有点像编译器和机器码。今天大多数人不需要理解编译器如何把代码变成机器指令，但这个世界仍然需要有人知道。Agent 时代也是一样。大多数人可能不再需要掌握底层每一行怎么写，但每个关键领域都需要少数真正懂 know-how 的人，去做审核、对齐、判断和修正。

可是顶尖专家不能靠强迫生产出来。一个人要成为专家，往往是因为他真的执着于这个领域。当大部分人只在 Agent 层面操作时，总要有人愿意下到更底层，去理解系统的真实结构。

其他大多数人的工作，可能会更多转向 creativity、判断、表达、沟通、体验和关怀。执行越来越抽象以后，主观体验和 taste 会变得更重要。做产品、做软件，某种意义上会更像艺术创作：你要说清楚你想要的世界和别人有什么不一样。

## Agent 的 OS 可能回到 Unix 哲学

我很喜欢一个观察：Agent 时代反而让我们重新理解 Unix 哲学。

Unix 里有一个精神叫 everything is file。能用文本表达的，就优先用文本。输入输出是文件，协议尽量可读，工具之间通过明确的接口协作。

现在很多 Agent 工作流也是这样。你给它一个 workspace，里面有 rules、memory、SOP、examples、rare cases、风格偏好、历史输出，它读完这个目录，就知道自己应该怎么做。

这也是为什么 rules 文件、CLAUDE.md、skills、workflow 文档会变得重要。它们不是普通说明书，而是 Agent 能读取的组织记忆。一个人的风格、一个团队的 SOP、一个项目处理极端情况的方法，都可以被 document 化。

但这里也有新的问题。如果公司开始记录你怎么和 AI 互动，记录你的 prompt、判断和风格，它其实也在学习你。未来招工程师，也许不只看你现场写不写得出算法题，而是看你怎么和 AI 协作，怎么审查结果，怎么留下可复用的规则。

## 为什么十倍效率还没有变成十倍产出

很多人问：既然 AI 把个人效率提高了这么多，为什么公司产出还没有十倍？

我的答案是：因为组织不是十倍的。

个人可以变快，但组织结构仍然慢。沟通、审批、层级、抽象、交接、对齐，都会消耗掉效率。传统公司一层一层传递指令，每一层都要理解、证明、汇报，再往下传。

Agent 带来的变化，是它可以穿透很多中间层。你可以让多个 Agent 像 swarm 一样并行工作，分别负责 research、实现、测试、review、汇报。限制不再只是模型能不能做，而是 token、预算、规则和你的判断力够不够。

所以组织会变扁平。以前五六层，未来可能两三层，甚至两层就够了。负责人定义目标，下面是一组 Agent 和少数人类治理者。

那人在公司里做什么？短期内，人会负责几个板块的治理、现实世界的接触、体验、沟通和判断。你负责一个系统的治理，决定它的规则、边界、价值观和长期方向。

## 软件会从标准化走向定制化

今天的 SaaS 很标准化。大公司可以做大量 internal tools，但十几二十人的小公司只能用标准 SaaS，因为他们没有能力养一支团队定制系统。

这个前提正在变。未来任何一个小团队，都可能让 Agent 按自己的需求生成软件。工具不一定是“买来长期适配”，而是“需要什么就写什么，用完继续演化”。

所以我同意一个判断：任何公司都会变成软件公司，任何业务都会变成软件业务。

公司形态也可能两极化。一边是一个人或几个人的小公司，靠 Agent 放大能力；另一边是巨型公司，掌握基础模型、算力、分发、资本和基础设施。

这不一定是好事。效率会进去，因为它太强了。但社会要承受它的副作用。一个可能的解法，是提高生存底线，让更多人不被迫用工作证明自己的生存价值。

## 我最后真正担心的是判断力

我不担心人类不会再写代码。总会有人写，总会有人钻到底层，总会有人因为执着而成为专家。

我更担心的是，我们以为自己还会判断。

当 AI 可以帮我们写代码、做产品、生成工具、协调 Agent、渲染汇报、处理数据，我们很容易把“结果看起来像样”误认为“结果是好的”。但一个东西短期有用，不代表长期无害；指标漂亮，不代表生态健康；AI 回答流畅，不代表它知道自己在说什么。

Agent 时代的程序员，可能不再只是程序员。他更像系统设计师、价值定义者、规则制定者和治理者。这个角色听起来更大，也更重。

因为真正稀缺的，不是能不能让机器写出东西。

真正稀缺的是：当机器写出东西以后，我还知不知道它好不好。
