# Video2Blog 桌面端 · 高稳定性技术方案

> 状态：设计稿（待评审）
> 最近更新：2026-05-25
> 关键决策：① Step 3–8 **运行时读 `WORKFLOW.md` 合同**（不在代码里重写规则）；② Step 3–8 **直连 LLM API**（按 token 计费，自建 prompt 装配）。

---

## 0. 设计立场

桌面端最大的不稳定源是两个外部黑盒：**Metal 转录**（慢、可能 OOM/超时）和 **LLM API**（慢、会限流、输出不确定、要花钱）。本方案不试图消除黑盒，而是把它们包进一个**纯检查点架构**：

> 每一步都是 `读磁盘产物 → 处理 → 原子落盘` 的幂等函数，全部状态都在磁盘上。任何环节崩溃、断网、关机，都能从上一个检查点续跑，绝不从头再来、绝不重复烧钱。

这套思路天然契合本仓库——它本来就是按产物分层的（`raw.txt` → `clean.md` → `Posts/*.md`），我们只是把这个隐含结构升级成显式的状态机。

## 1. 三层架构（职责不重叠）

| 层                     | 内容                                                                                              | 改动                                  |
| ---------------------- | ------------------------------------------------------------------------------------------------- | ------------------------------------- |
| **合同层**       | `WORKFLOW.md` + `.cursor/skills/<step>/SKILL.md` + `knowledge/STYLE_GUIDE.md` + `memory/` | **不动**，仍是唯一真相          |
| **引擎层**（新） | 运行时解释器：装配上下文 → 调 API → 质检 → 落盘                                                | 新写，但近一半骨架可复用 `scripts/` |
| **外壳层**（新） | FastAPI 本地服务 + UI                                                                             | MVP 用浏览器，原生壳推迟              |

```
UI (浏览器/原生)
   │  SSE 流式 + REST
FastAPI 本地服务  ──状态机──►  work/<stem>/.state.json   ← 崩溃恢复的唯一依据
   │
引擎 Engine
   ├─ ContextLoader   读合同（按 WORKFLOW.md §2/§4 解析，不硬编码文件名）
   ├─ StepRunner      通用步骤循环（幂等 + 重试 + 原子写）
   ├─ LLMClient       直连 API（重试/退避/超时/成本计）
   └─ Guards          Pre-Flight / 合同校验 / 成本护栏 / 视角扫描
   │
Step 1-2: pipeline.py（复用）→ ffmpeg + mlx/whisper.cpp（子进程隔离）
Step 3-8: 合同驱动 → LLM API
```

## 2. 稳定性核心机制（方案的重心）

### 2.1 步骤即检查点 · 断点续跑

- 每个 job 在 `work/<stem>/.state.json` 维护状态机（含 §9 的增强态）：
  ```
  full:  PENDING → TRANSCRIBING → CLEANED → EXTRACTED → STRUCTURED
             → [WAITING_USER_OUTLINE] → REWRITING ⇄ CHECKING(自修正≤N)
             → PASS:DONE  |  REVIEW:WAITING_USER_REVIEW → DRAFT_DONE
  quick: PENDING → REWRITING ⇄ CHECKING(自修正≤N) → DONE | WAITING_USER_REVIEW
  ```

  （quick 模式跳过 EXTRACTED/STRUCTURED 等中间态）。`REWRITING ⇄ CHECKING` 是 §9-A 的自我修正闭环；`WAITING_USER_*` 是 §9-B 的人工节点。
- 每步完成后**先落产物、再更新 state**（顺序不能反，保证 state 永远不超前于磁盘）。
- 引擎启动先读 state，从最后一个完成步骤的下一步续跑。**关机重开、API 中途失败，都不丢前面的成果。**

### 2.2 幂等与去重（既保稳定也省钱）

每步用 `(输入产物 hash + step + 模型 + 合同版本)` 做键。键没变就复用已有产物、跳过 API 调用。带来三重收益：可重跑、可复现、不重复计费。

### 2.3 LLM 输出可控 + 质检闸门

- **温度固定、模型版本钉死**（写进 state 和文章 frontmatter，可审计「哪个模型 + 哪版合同产出了这篇」）。
- **Step 7 结构化判定**：复用仓库已有的评分契约——`WORKFLOW.md` Step 7 正好 6 个维度（忠实度 / 可读性 / 观点密度 / 风格一致 / 完整性 / 视角忠实度），`scripts/validate_workflow.py` 的 `PASS_SCORE_RE` 已定义 `x/60`。让 LLM 返回结构化 JSON：
  ```json
  {"verdict":"PASS|REVIEW","scores":{"忠实度":9,"可读性":9,"观点密度":8,"风格一致":9,"完整性":9,"视角忠实度":10},"total":"54/60","rebrief":"..."}
  ```

  6 维 ×10 = 60，无缝对接现有 Review 格式。
- **REVIEW 绝不自动发布**。PASS → Step 8 正文；REVIEW → 停在人工确认，用户明确接受才以 `DRAFT-` 落盘。这是产品级硬闸门。
- **自动视角扫描**：复用 `validate_workflow.py` 的 `VIEWER_RE`，每篇草稿落盘前自动查「观看者 / 编者视角」违规词——零成本质量兜底。

### 2.4 合同完整性守卫

- 引擎启动先跑**合同校验**（扩展现有 `validate_workflow.py` + `check_placeholders`）：占位符未填、skill 文件缺失、frontmatter schema 不符 → **拒绝运行并明确报错**，而不是带病产出。
- **合同指纹**：把 `WORKFLOW.md + skills + STYLE_GUIDE + PREFERENCES` 哈希成一个版本号，写进 state 与文章 frontmatter。合同一改，相关 job 缓存自动失效——避免「旧合同缓存污染新产出」。

### 2.5 成本与上下文护栏

- **预估闸门**：每个 job 开跑前先估算 token 与费用，超过用户设定上限就先弹确认。
- **硬上限**：单 job token 预算上限，超了立即中止（防 prompt 循环烧钱）。
- **长稿分块策略**（真工程，不能省）：
  - Step 3 清洗 / Step 4 提炼：**map**（分块独立处理）→ **reduce**（合并），因为这两步是局部的。
  - Step 5 结构 / Step 6 改写：需要全局视野 → **先出大纲，再按节逐段改写并对齐大纲**，避免单次喂全文超窗。

### 2.6 数据安全

- **原始 ASR 只追加不覆盖**（仓库铁律，引擎强制执行）。
- **原子写**：所有落盘走 `临时文件 + os.replace`（同盘原子），杜绝写一半崩溃留下半个文件。当前 `output.py` / `pipeline.py` 是直接 `write_text`，本方案统一升级为原子写 helper。
- **成品覆盖保护**：`output/Posts/` 同名先备份再写；沿用 `-v2/-v3` 冲突策略。
- 严守 `input/`（原料）/ `work/`（过程）/ `output/`（成品）三条边界。
- **API Key 不落盘**：多配置档，每档 key 经 `keyring` 存 macOS 系统钥匙串；日志/错误/任务对象全程脱敏（`***`），`GET /api/llm-profiles` 只回末四位与来源。**【已落地 2026-05】**

### 2.7 可观测性

- 每步结构化日志（沿用 `raw.log` 风格），失败可定位到具体 step + API 请求。
- UI 折叠控制台实时回放日志。

## 8. 设计支点小结

高稳定性落在三个支点上：

1. **检查点架构**（断点续跑，不丢成果、不重复烧钱）；

## 9. 引擎增强机制（A–D）

四个机制用纯 API 模拟出 Agent 的交互能力，每个都已按稳定性原则做过修正。

### A. 自我修正闭环（Self-Correction Loop）

**目标**：用纯 API 模拟 Agent 的反复纠错，提升一次性出稿质量。
**做法**：StepRunner 将 Step 6（重写）与 Step 7（质检）级联成 `REWRITING ⇄ CHECKING` 循环：
生成初稿 → 跑质检 → 不过则把质检反馈喂回、追加修正提示重写 → 至多 N 轮。

稳定性修正（必须遵守）：

1. **择优而非取末轮**：保留每轮草稿与分数，最终选「总分最高且通过 `VIEWER_RE`」那版；防止越修越差还照发。
2. **免费检查前置**：先跑确定性的 `VIEWER_RE`（零成本），通过才花钱调 Step 7 LLM 评分；不通过直接短路进修正。
3. **重试计入预算**：1 次重试 ≈ 多 1×Step6 + 1×Step7（最坏 3× 开销），必须计入 §2.5 token 硬上限；最大轮次可配，默认 1 次重试。
4. **修正提示词归合同所有**：二轮润色的提示模板写进 `rewrite-blog` skill，由引擎注入（见 D），不硬编码进引擎，避免规则两处漂移。
5. **达上限仍不过 → 转 REVIEW 交人工**，绝不静默发布更差稿；轮次与分数落进 `.state.json`，崩溃可续。
6. （可选）重试时可微调温度以逃离同一失败模式，但需受控。

### B. 半自动审批节点（Human-in-the-Loop）

**目标**：把大纲（Step 5）的控制权交还创作者，消除"失控感"。
**做法**：状态机增挂起态 `WAITING_USER_OUTLINE`——Step 5 完成后暂停，UI 渲染大纲树供拖拽重排/改标题/增删，用户「确认并重写」后将最终大纲写入 `work/<stem>/outline.md` 并唤醒 Step 6。

稳定性修正：

1. **仅 full 模式**（quick 无 Step 5）；quick 可另设可选 `WAITING_USER_DRAFT` 复核点。
2. **暂停是 per-job 开关**（full 默认开），想一键全自动可关。
3. **产物名用 `outline.md`**，避免与 Step 3 的 `clean.md` 混淆（原提案 `clean_outline.md` 易误读）。
4. 用户改完的大纲是不可变检查点；改大纲改变 Step 6 输入 hash，**自动失效下游缓存**（与 §2.2 幂等一致）。
5. UI 须显式列出「等待你输入」的挂起 job，防遗忘。

### C. 分块的滑动窗口上下文（Sliding Window Context）

**目标**：超长视频做局部清洗/改写时，消除切片边缘的转折突兀、指代不明。
**做法 + 修正**：

1. **清洗用「上下文重叠」而非「输出重叠」**：模型**只读**前块尾部做衔接，**只输出**新区段——零重复、无需去重。（原提案的 10% 输出重叠需去重、易碎，已弃。）
2. **确定性 + 句子边界分块**：复用 `transcript.py` 断句逻辑与 `update_fingerprint.py` 的 `SENTENCE_RE`，保证同输入同分块，幂等键稳定。
3. **改写滑动窗口**：每节改写时附带「全局大纲树」+「上一节已成稿的末段」；滚动摘要按需再生成（注意其 token 成本）。

### D. Skill Markdown 标准解析器

**目标**：让 ContextLoader 优雅读取 `.cursor/skills/<step>/SKILL.md` 并组装任务指令。
**做法 + 修正**：

1. **复用现有 PyYAML `strip_frontmatter`**（`validate_workflow.py`）剥离 frontmatter，**不写第二个 YAML 解析器**——避免两处漂移。
2. **结构化拼装为主、`{{VAR}}` 为辅**：大块内容（合同+skill+风格+memory → system，产物正文 → user）靠消息结构拼；`{{VAR}}` 仅用于少量动态小字段（SPEAKER/ROUTING/MODE/SOURCE）。
3. **变量 fail-closed**：未知或未填变量直接报错，绝不把字面量 `{{SOURCE}}` 发给 API（接 §2.4 合同守卫）。
4. 解析结果按**合同指纹**缓存（§2.4）。
