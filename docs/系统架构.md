```
         │

│  video2blog/server.py        装配：FastAPI + CORS + lifespan  │

│  WORKFLOW.md                  唯一权威：入口/模式/路由/产物 │

│  knowledge/STYLE_GUIDE.md     文风硬约束                    │
│  knowledge/Examples/          few-shot 范文                  │
│  memory/PREFERENCES.md        语言/人称/禁用表达             │
│  memory/CONFIG.md             路径/ASR 引擎/输出约定          │
│  memory/HISTORY.md            最近 10 篇索引                  │

└──────────────────────────────────────────────────────────────┘
```

**关键约束：合同层不嵌入代码**。引擎运行时去读这些 .md 文件、解析 frontmatter 与正文，装到 LLM 上下文里。合同改了 → 哈希变了 → 旧缓存自动失效。

## 2. 数据流（一次 full 模式 video job 的完整路径）

```
input/Video/foo.mp4
   │
   │  video2blog/pipeline.py  subprocess 调 video2blog.py
   ▼
work/foo/raw.txt   raw.srt   raw.log   meta.json    ← Step 1–2
   │
   │  engine.runner._run_clean_full
   ▼
work/foo/clean.md                                    ← Step 3
   │
   │  engine.runner._run_extract_full
   ▼
work/foo/insights.md   (state.cache["EXTRACTING"])  ← Step 4
   │
   │  engine.runner._run_structure_full
   ▼
work/foo/outline.md                                  ← Step 5
   │
   │  ─── 暂停：WAITING_USER_OUTLINE ───
   │  前端 OutlineView 编辑 + POST /jobs/{id}/approve-outline
   ▼
work/foo/outline.md（用户改后版）
   │
   │  engine.runner._run_rewrite  支持 single / sectioned
   ▼
work/foo/draft_v1.md  …  draft_vN.md（自修正多版本）  ← Step 6
   │
   │  engine.runner._run_check    6 维 × 10 = 60
   ▼
work/foo/review_v1.md  …  review_vN.md               ← Step 7
   │
   │  择优：总分最高 + 视角忠实度 > 5 + VIEWER_RE 零违规
   │
   ├─ PASS  ──▶ output/Posts/2026/<date>-<标题>.md   ← Step 8
   │           output/Reviews/<date>-<标题>.review.md
   │           memory/HISTORY.md（追加）
   │           memory/fingerprints.jsonl（追加）
   │
   └─ REVIEW ─▶ WAITING_USER_REVIEW
                前端 DraftReviewView 编辑 + 接受
                POST /jobs/{id}/approve-draft
                output/Posts/2026/DRAFT-<date>-<标题>.md
```

## 3. 状态机

`work/<stem>/.state.json` 持久化，引擎冷启动从这里恢复。

```
                                  ┌─────────────────┐
PENDING ──┬─full──▶ CLEANING ────▶│  EXTRACTING     │──▶ STRUCTURING
          │                       └─────────────────┘             │
          │                                                       ▼
          │                                            WAITING_USER_OUTLINE
          │                                                       │
          │                                                       │ approve
          │                                                       ▼
          └─quick──────────────────────────────────────▶  REWRITING ⇄ CHECKING
                                                                  │  （自修正 ≤ N 轮，
                                                                  │   择优而非取末轮）
                                                                  ▼
                                                          ┌───────┴───────┐
                                                          ▼               ▼
                                                  PASS:FINISHED   REVIEW:WAITING_USER_REVIEW
                                                                          │
                                                                          ▼
                                                                    DRAFT_DONE / 用户取消
```

终态：`FINISHED` / `DRAFT_DONE` / `FAILED` / `CANCELLED`。重新提交同 stem 时全部重置回 `PENDING` 并清旧 Step 6/7 产物。

## 4. 后端 API 表面（FastAPI · localhost:8765）

按职能分四块：

### 4.1 健康 / LLM 配置
- `GET /health`
- `POST /api/test-llm`
- `GET / POST / PUT / DELETE /api/llm-profiles[/{id}]`
- `POST /api/llm-profiles/{id}/default`
- `DELETE /api/llm-profiles/{id}/key`

### 4.2 素材 / 任务
- `GET /sources`（同时列 `input/Text/*` 和 `input/Video/*`）
- `POST /upload`
- `POST /api/detect-speaker`
- `POST /jobs`（建任务，含 mode / routing / strategy / profile_id 等）
- `GET /jobs` `/jobs/{id}` `/jobs/history`
- `GET /jobs/{id}/events`（SSE 实时事件流，见 §4.5）
- `POST /jobs/{id}/approve-outline` `/approve-draft` `/cancel`

#### 4.5 SSE 事件协议（H1 去耦后）

job 事件流有两类载荷，职责分离：

| 事件 | 载荷 | 用途 |
|---|---|---|
| `log` | `{message}` 原始 print 文本 | 「原始日志」视图逐行排查 |
| `progress` | `{kind, step?, version?, verdict?, total?, path?, …}` 语义字段 | 前端叙事 + StepProgress |
| `queued/started/paused/succeeded/failed` | job 生命周期 | 状态跃迁、提醒 |

`progress.kind` 词表：`job_start` / `step` / `verdict` / `self_correct` / `viewer_blocked` / `parse_failed` / `max_retries` / `artifact` / `transcribe`。

**语义在后端、展示在前端**：`runner._progress()` 与相邻 print 同处定义（print 给人看、progress 给机器用，不漂移）；前端 `log-parser.mapProgress()` 只做 kind→文案 的映射。转录前三步的子进程 stdout 标记（`[1/3]` 等）在 `server_core._transcribe` 唯一一处翻成 `transcribe` 进度事件，前端彻底不碰正则。

> **已知债**：SSE 手动重连时 `iter_events` 从 cursor 0 全量重放，前端会重复追加事件（`log` 与 `progress` 同此行为，H1 前后一致）。根治需重连按 `Last-Event-ID` 续传，列入架构债。

### 4.3 产物
- `GET /artifacts/{id}`
- `GET /jobs/{id}/files/{file_key}`
- `GET /work-files`
- `GET /file?path=...`（受限只读 output/ 与 work/）
- `POST /open`（NSWorkspace 打开 Finder / 默认编辑器）

### 4.4 知识库可视化编辑
- `GET /knowledge-files`（分组列举：写作偏好/风格与范文/运行合同·路由/步骤提示词/系统/参考范文）
- `GET /knowledge-file?path=...`
- `PUT /knowledge-file`（atomic_write + 占位符/WORKFLOW 行数轻校验，非阻塞返回 `{ok,errors}`；白名单外 → 403）

### 4.6 你的声音（文风画像）
- `GET /fingerprints`（聚合 `memory/fingerprints.jsonl` → count/平均句长/平均段长/每篇句长走势/跨篇高频词；只读纯本地，`routes/voice.py`）

## 6. 引擎核心机制（A–D · 对应桌面端方案 §9）

| 机制 | 当前位置 | 状态 |
|---|---|---|
| A · 自我修正闭环（REWRITING ⇄ CHECKING ≤ N） | `runner.py` `_run_rewrite_check_loop` | ✅ 已落地，择优、计入预算、达上限转 REVIEW |
| B · 大纲审批节点 WAITING_USER_OUTLINE | `runner.py` + `POST /approve-outline` + 前端 OutlineView | ✅ full 模式默认开 |
| C · 长稿按节滚动（sectioned）+ 滑动窗口上下文 | `engine/chunking.py` + `outline.py` + `runner.py` 的 sectioned 分支 | ✅ 解析失败自动回退 single |
| D · Skill Markdown 标准解析器 | `engine/parser.py`（复用 `validate_workflow.strip_frontmatter`） | ✅ 变量 fail-closed、合同指纹缓存 |

## 7. 已落地的稳定性保障

| 关键点 | 实现位置 |
|---|---|
| 原子写 helper | `engine/utils.atomic_write` |
| 检查点 .state.json + 顺序"先落产物再写 state" | `runner.py` `_record_artifact` |
| 幂等键 (input hash + step + model + 合同指纹) | `runner.py` `cache_key` |
| 合同指纹 + 一改全失效旧缓存 | `parser.ContextLoader.compute_fingerprint` |
| `VIEWER_RE` 视角扫描 | `video2blog/utils.py` + Step 7 兜底 |
| 6 维 × 10 评分解析 | `runner.parse_markdown_review` |
| 子进程隔离的转录前置 | `_run_job` 用 subprocess 调 `video2blog.py --no-auto-terminal` |
| 取消 + wall-clock 超时（`VIDEO2BLOG_TRANSCRIBE_DEADLINE`） | `_run_job` 后台线程读 stdout |
| Key 全程脱敏（任务对象 / SSE / 错误） | `secrets_store.redact` |
| 本地服务安全边界（仅 localhost、source 仅仓库根内） | `server.py` startup |
| 故障注入回归（mock LLM 跑 5 个金标 fixture） | `scripts/regression.py` + `tests/fixtures/regression/` |

## 8. Phase 3 / 下一阶段未完事项

按"卡用户最重的事"排序：

1. **Tauri sidecar 集成**：把 `make backend-bin` 出来的 `video2blog-server` 声明为 `tauri.conf.json` 的 `externalBin`，主进程启动时拉起 / 退出时回收。痛点是未签名二进制访问钥匙串会弹授权，curl 等不到。
2. **Apple Developer ID 签名 + Notarization**：CI 跑 `codesign` + `notarytool submit --wait`，entitlements（network.client / files.user-selected）写齐。
3. **`make app-build` 接住一切**：补 `make backend-bin` → 签名 → 公证 → `.dmg`。
4. **whisper.cpp bundle**：分发版默认 bundle whisper.cpp + ggml，MLX 留给高级用户（绕开 `.metallib` 打包大坑）。
5. **golden-set 回归扩展**：现有 5 个 fixture 太少，把 48 篇成品做成更大的 baseline。
6. **可观测仪表盘**：把 .state.json 里的 token/费用/wall-clock 聚合到 FinalView，让用户看见单篇成本。
7. **自动更新通道**（可选）：tauri-updater + GitHub Releases，签名 manifest。

## 9. 主要风险（工程视角）

| 风险 | 影响 | 当前对策 |
|---|---|---|
| `mlx-whisper` 的 `.metallib` 打不进 PyInstaller | 阻塞 sidecar | 本轮排除 mlx，先走 whisper.cpp |
| 未签名二进制弹钥匙串授权 | 首启卡死 | 必须先解决 Developer ID 签名 |
| 长稿 sectioned 时 LLM 把大纲改没了 | 跑飞 | parser 检测到骨架不可识别自动回退 single |
| 自修正死循环烧钱 | 账单爆 | 默认 N=1 + 单 job token 硬上限 |
| 合同被改坏（占位符未填、frontmatter schema 错） | 带病产出 | 引擎启动前 `validate_workflow.py` 校验，命中 STOP |
| 用户改 STYLE_GUIDE 后旧文件不一致 | 风格漂 | 合同指纹写进 frontmatter，可审计；不强制重写历史 |

## 10. 一句话总结

**合同层是规则，引擎层是检查点状态机，外壳层是 Claude 桌面端品牌气质的本地 app**。三层各自独立、改动互不污染，是这个项目能稳定迭代到 48 篇成品的根本原因。下一步主要补的是"打包分发"和"质量基线沉淀"这两块工程化基建，而不是再加新功能。
