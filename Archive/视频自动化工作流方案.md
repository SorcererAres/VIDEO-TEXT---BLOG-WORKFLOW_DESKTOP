# 视频内容自动化生产方案

> **【已归档 · 设计背景】** 工程选型与风险记录，不作运行依据；当前运行约定见 `knowledge/工作流契约.md` 与 `项目结构.md`。

> 从本地视频文件自动生成字幕与第一人称博文的端到端工作流

---

## 1. 背景与目标

### 1.1 现状

日常会积累一批视频分享文件(讲座、对谈、录屏分享等),通常具备以下特征:

- 源文件多为本地 `.mp4` / `.mov` 等格式,无字幕轨
- 语言为中英混合(中文为主,夹杂英文术语)
- 时长以 30–60 分钟为多
- 每周需处理数次

### 1.2 痛点

- 视频内容不可检索、不可摘要,沉淀价值低
- 手动整理博文耗时,且转录环节是瓶颈
- 现有零散工具(在线转录 + 手动复制 + 手动改写)割裂,无法批量处理

### 1.3 目标

搭建一套**全自动本地工作流**,实现:

1. 视频文件投入指定文件夹后,无需人工干预
2. 自动生成标准 `.srt` 字幕(可挂回原视频使用)
3. 自动生成第一人称博文 Markdown(可直接发布或入库 Obsidian)
4. 处理质量稳定,中英混合识别准确
5. 运行成本可控(单条视频成本以美分计)

## 2. 总体架构

### 2.1 流水线

```
┌────────────────┐
│  视频投递目录   │  ← 用户拖入视频文件
│  ~/Movies/inbox│
└───────┬────────┘
        │ watchdog 监听
        ▼
┌────────────────┐
│  ffmpeg 抽音频  │  → 16kHz 单声道 wav
└───────┬────────┘
        ▼
┌────────────────┐
│  mlx-whisper   │  → 中英混合自动识别
│  large-v3      │
└───────┬────────┘
        │
        ├──→ output/xxx.srt   (带时间戳字幕)
        ├──→ output/xxx.txt   (纯文本转录)
        ▼
┌────────────────┐
│  Claude API    │  → 第一人称博文改写
│  Sonnet 4.5    │
└───────┬────────┘
        ▼
┌────────────────┐
│  output/xxx.md │  → Obsidian / Notion 可读
└────────────────┘
```

### 2.2 分阶段实施

| 阶段 | 形态 | 目标 |
|---|---|---|
| Phase 1 | 单 Python 脚本 | 跑通流程,验证质量 |
| Phase 2 | n8n 可视化工作流 | 加上通知、归档、错误重试 |
| Phase 3 | (可选) 进阶能力 | 长视频分段、关键帧截图、自动入库 |

本方案聚焦 **Phase 1**,Phase 2/3 在第 7 节展望。

## 3. 技术选型

### 3.1 运行环境

- **硬件**: Apple Silicon Mac (M1/M2/M3/M4)
- **OS**: macOS 13+
- **Python**: 3.10+

### 3.2 关键组件

| 环节 | 工具 | 选型理由 |
|---|---|---|
| 音频提取 | ffmpeg | 行业标准,稳定 |
| 语音转录 | **mlx-whisper** + `large-v3` | Apple Silicon 上比 faster-whisper 快 2–3 倍,显存占用低;`large-v3` 是当前中英混合识别最强模型 |
| 大模型改写 | **Claude Sonnet 4.5** | 中文写作质量高、可控性强;成本相比 Opus 低一个数量级,日常使用够用 |
| 文件夹监听 | watchdog | Python 生态成熟方案 |
| 配置管理 | python-dotenv | 简单可靠 |

### 3.3 为什么不选其他方案

- **不选在线转录(飞书妙记 / 通义听悟)**: 隐私上不可控,无法编程接入,批处理体验差
- **不选 OpenAI Whisper 官方包**: 在 Mac 上比 mlx-whisper 慢
- **不选 faster-whisper**: 在 Apple Silicon 上 mlx-whisper 性能更优
- **不选 GPT / Gemini 做改写**: Claude 在中文长文本改写、保持人称一致性上目前表现更稳

## 4. 详细设计

### 4.1 目录结构

```
~/Movies/
├── inbox/                  # 用户投递视频
│   ├── 视频A.mp4
│   ├── 视频B.mov
│   └── output/             # 自动生成产物
│       ├── 视频A.srt
│       ├── 视频A.txt
│       ├── 视频A.md
│       └── ...
└── ...
```

### 4.2 处理流程详解

**Step 1 — 音频提取**

```
ffmpeg -i video.mp4 -ac 1 -ar 16000 -vn audio.wav
```

- 单声道 (`-ac 1`) + 16kHz (`-ar 16000`):Whisper 训练时的目标格式
- 跳过视频流 (`-vn`):只要音频,节省 IO

**Step 2 — 语音转录**

调用 mlx-whisper 时的关键参数:

| 参数 | 取值 | 作用 |
|---|---|---|
| `language` | 不指定 | 让模型自动判断每段语言,中英混合的关键 |
| `condition_on_previous_text` | `False` | 防止幻觉累积扩散 |
| `no_speech_threshold` | `0.6` | 静音段过滤,减少瞎编字幕 |
| `word_timestamps` | `False` | 提速,博文场景不需要词级时间戳 |

**Step 3 — 字幕与纯文本生成**

- `.srt`: 标准字幕格式,可直接挂回视频
- `.txt`: 简单清洗(去多余空白、按句号断行),作为大模型输入

**Step 4 — 博文改写**

通过 Claude API 调用,核心是 **系统提示词**:

```
- 第一人称叙述,口吻自然
- 介于口语与书面之间:有观点、有逻辑,保留对话感
- 删除口头禅、重复、跑题
- 保留观点、案例、数据
- 专业术语保留英文原词
- 不要"大家好"、"以上就是"等视频套话
- 输出纯 Markdown,无元信息
```

**长稿处理**: 单次请求上限 24,000 字,Phase 1 直接截断兜底;Phase 3 升级为分段改写+合并。

### 4.3 配置项

通过 `~/.video2blog.env` 管理:

```
ANTHROPIC_API_KEY=sk-ant-xxx
```

模型选择、风格 prompt 在脚本顶部常量区调整,便于迭代。

## 5. 部署与使用

### 5.1 一次性环境准备

```bash
# 1. 安装 ffmpeg
brew install ffmpeg

# 2. 安装 Python 依赖
pip install mlx-whisper watchdog anthropic python-dotenv

# 3. 配置 API Key
echo 'ANTHROPIC_API_KEY=sk-ant-xxx' > ~/.video2blog.env
```

首次运行会自动下载 `large-v3` 模型(约 3GB),后续走本地缓存。

### 5.2 两种使用模式

**单文件处理(临时)**

```bash
python video2blog.py ~/Movies/某次分享.mp4
```

**常驻监听(推荐)**

```bash
python video2blog.py --watch ~/Movies/inbox
```

启动后保持运行(可用 `tmux` 或 macOS `launchd` 后台化),拖视频进去即自动处理。

### 5.3 产物说明

每个视频生成三个文件,放在 `inbox/output/` 下:

| 文件 | 用途 |
|---|---|
| `xxx.srt` | 标准字幕,挂回原视频用 |
| `xxx.txt` | 纯文本转录,做检索或归档 |
| `xxx.md` | 第一人称博文,直接发布或入库 |

## 6. 成本与性能

### 6.1 时间开销(M2 Max 实测参考)

| 视频时长 | 抽音频 | 转录 (large-v3) | 改写 (Sonnet 4.5) | 总计 |
|---|---|---|---|---|
| 10 分钟 | ~5s | ~1 分钟 | ~30s | ~2 分钟 |
| 30 分钟 | ~15s | ~3 分钟 | ~1 分钟 | ~5 分钟 |
| 60 分钟 | ~30s | ~6 分钟 | ~1.5 分钟 | ~8 分钟 |

转录占大头,但全程无需人工。

### 6.2 金钱成本

转录在本地完成,**零成本**。改写费用 = Claude API 调用费:

- Sonnet 4.5: 单条 30 分钟视频约 **¥0.1–0.3 RMB**
- Opus 4.5: 单条 30 分钟视频约 **¥1–2 RMB**

按每周 3–5 条估算,月度成本可忽略。

### 6.3 隐私

- 视频与音频 **100% 本地处理**,不上传任何云端
- 仅"转录文本"被传给 Anthropic API 用于改写
- 如对内容隐私敏感,可改为本地 LLM(如 Qwen / Llama)做改写,但中文质量目前不及 Claude

## 7. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 中英混合识别错误 | 字幕质量下降 | 可通过 `initial_prompt` 引导模型(如"以下是一段中英混合的技术分享") |
| 静音段幻觉 | 字幕出现无中生有内容 | 已启用 `condition_on_previous_text=False` + 高 `no_speech_threshold`,基本可控 |
| 博文风格不符预期 | 输出风格偏离 | 提示词在脚本中央常量化,易于迭代;先跑 2–3 条样本回调风格 |
| 长视频(>1小时)超出单次上下文 | 改写被截断 | Phase 1 提示警告 + 截断;Phase 3 升级分段策略 |
| API Key 泄露 | 账单风险 | 存在 `~/.video2blog.env`,文件权限 `chmod 600` |
| 监听误触发(文件未写完就开始处理) | 处理失败 | 已实现"文件大小稳定 3 次"判定机制 |

## 8. 演进路线

### Phase 2 — n8n 可视化工作流(预计 1 天)

将单脚本拆为 n8n 节点,新增能力:

- 失败自动重试 (3 次)
- 处理结果推送到 Telegram / 企业微信
- 自动归档到 Notion / Obsidian Vault
- 多文件夹监听 + 路由(根据来源选不同 prompt)

### Phase 3 — 进阶能力(按需)

- **长视频分段改写**: 按章节切分 → 并行改写 → LLM 合并
- **关键帧截图**: 自动抽取 PPT 切换帧,嵌入博文
- **多风格 prompt**: 维护 prompt 库,根据视频类型自动路由
- **质量自评**: 用 Claude 反向打分,低于阈值的产物进人工复核队列
- **多格式输出**: 同步生成微信公众号格式 / 知乎格式 / Newsletter 格式

## 附录 B:常见问题

**Q: 第一次运行卡在"转录中"?**
A: 在下载 large-v3 模型,约 3GB,首次需联网。后续走本地缓存。

**Q: 改写出来的博文风格不喜欢?**
A: 改脚本顶部 `BLOG_SYSTEM_PROMPT`,这是控制风格的唯一旋钮。

**Q: 想换成 GPT / Gemini 改写?**
A: 把 `rewrite_to_blog()` 里换成对应 SDK 即可,接口形态类似。

**Q: 想处理 1 小时以上的视频?**
A: 当前版本会截断到 24,000 字。需要分段改写能力的话,告知即可加。
